01跨 session 失忆

Agent 最痛的问题，不致命但持续消耗注意力
  • 下午花几小时让 agent 理解一个 checkpoint 保存 bug，翻了几十层调用栈
  • 第二天打开新对话窗口继续，agent 一脸茫然地看着 checkpoint 路径里的报错
  • 中间那个关键的 epiphany（"啊原来这条路径绕过了 validation"）——过程已经丢失

03架构总览：四角色分工

不是单一进程，是四个角色协同

04为什么不是简单的 MCP server

因为 memory 写入是高频且不能阻塞对话的
  • 朴素设计：写一个 MCP server，提供 read/write 工具，agent 想用就调
  • 问题：每次工具调用都触发一次写入，AI 压缩需要 5-30 秒
    → Claude Code 的工具调用被卡住 5-30 秒
  • claude-mem 的方案：写入做成异步管道
    • PostToolUse hook 同步触发 → worker 收到 webhook
    • worker 原子地往 pending_messages 写一行 + 入队 job
    • worker 后台 job runner 慢慢调 AI 压缩
    • Claude Code 立即继续，不等 AI 完成

05Hook 生命周期

6 个事件回调覆盖 Claude Code session 整个生命周期
Session 启动 ├─→ SessionStart: 启动 worker（如未跑） ├─→ SessionStart: 查询历史 obs → 注入 system prompt ▼ 用户发消息 ├─→ UserPromptSubmit: 写 sdk_sessions 行 ▼ Agent 推理 + 调工具 ├─→ PreToolUse(Read): 拉文件相关 obs ├─→ [工具执行] ├─→ PostToolUse: 入队 observation 生成任务（异步） ▼ Agent 回复完成 ├─→ Stop: 生成 session summary ▼ 下一轮 / Session 结束

06Observation：write-time 压缩的艺术

每次工具调用都触发一次 AI 压缩，把原始上下文变成结构化观察
  • 原始上下文：tool_name + tool_input + tool_response（可能数千 tokens）
  • 压缩后：type=bugfix/feature/decision + 摘要 + 关键文件
  • 存储：SQLite observations 表 + ChromaDB 向量
  • 注入时机：新 session 启动时，按 project + recency 召回 top-N
⚠ 反直觉发现：LoMoDe benchmark 上，最朴素的"把对话直接存进文件"方案打败了精心设计的 memory 系统

07本质的张力：六个未解难题

08能从 claude-mem 学到什么

  • 异步管道是 agent 系统的命脉——同步等 LLM 是体验灾难
  • Hook 生命周期比"agent 主动调用"更可靠——不依赖模型记得调
  • write-time 压缩省注入 token，但 LoMoDe 警示它可能不是最优
  • Provider 抽象是小聪明也是大坑——兼容性永远在追
