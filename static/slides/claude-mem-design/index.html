The Backlash · 反噬

memory 设计不好，会反噬

盲目加 memory 系统的 4 种失败模式——让你宁可它没有 memory。

4 Modes of Failure · 4 种反噬
01
Attention 挤占
注入无关上下文，挤占宝贵的 attention budget——模型被噪声淹没。
02
推理污染
低质量记忆反复塞给模型，把幻觉和误判固化成"事实"。
03
错误召回
在错误时刻召回错误内容，让 agent 越走越偏，错误叠加。
04
隐私泄露
静默把敏感信息写进共享存储——你不知道什么被记下了。

所以值得花时间拆一个生产级方案，看它怎么避开这些坑。

Design Trade-off · 异步管道
06 / 22
Why not just an MCP server?

因为 memory 写入不能阻塞对话

Before · 同步阻塞

MCP server · read/write

  • Agent 调 write_memory 工具
  • 等 AI 压缩完成（5–30 秒）
  • Claude Code 工具调用被卡住
  • Session 关闭 → 任务丢失
After · 异步管道

Worker Queue · ms 返回

  • Hook 同步入队 → 毫秒级返回
  • 后台 worker 异步消费 + AI 压缩
  • 失败可 retry · 跨 session 存活
  • 代价：worker 必须是独立 daemon

收益：不阻塞 · 可控 retry · 跨 session。代价：worker 启动 / 升级 / 状态管理。

09 / 22
Write-time vs Read-time.

核心设计哲学：写入时压缩

每次工具调用让 AI 把 raw transcript 压成 100–300 字结构化 observation。

Raw Transcript

Token 灾难

  • 全量对话回放，每条几 KB
  • 信噪比低 · 召回困难
  • 每次注入预算爆炸
  • 无法跨 session 索引
AI-compressed Observation

100–300 字结构化

  • type / concept / files / project 元数据
  • FTS + 向量双索引
  • 召回 top-50 一次性入 prompt
  • read-time token 大幅节省

前提假设：写入少、读取多——但这个假设在 Claude Code 场景下未必成立。 — 伏笔：S16 Letta Filesystem 会拷问这一点

11 / 22
DeepSeek-V4-Flash · thinking patch.

50% → 100% · 一个 patch

Before · 默认 thinking

推理模型把 token 全用在 reasoning

{ model: "deepseek-v4-flash",
  // no thinking control }
  • content 字段空 · 全用在 reasoning
  • 解析器拿到 "Empty response"
  • 4 次 SDK 调用 2 次失败 → 50%
After · thinking: disabled

关掉推理，让 token 进入 content

{ model: "deepseek-v4-flash",
  thinking: { type: "disabled" } }
  • 成功率 100% · 0 次失败
  • token 用量降一半
  • thinking-control API 正成标配

结构化输出场景需要关掉 thinking——reasoning 模型 + 工具调用 = 这条 patch 必备。

Three cousins · 边界在哪.

Memory = RAG over agent's own experience

技术同构，但语料 / 更新 / 失效模式都不同——最关键：Memory 失败会强化错误认知（cascading）。

Memory

自己的经历

  • 语料：agent 自身 obs
  • 更新：每次工具调用
  • 失效：cascading · 强化错误认知
  • 个性化：强 · 动态
RAG

外部语料

  • 语料：文档 / 知识库
  • 更新：人工驱动
  • 失效：单次答错 · 易纠
  • 个性化：无 · 共享
Fine-tuning

稳定知识

  • 语料：训练数据集
  • 更新：周期性 batch
  • 失效：模型权重固化
  • 个性化：弱 · 静态

趋势：base model 进步会让 Memory 相对价值上升——RAG / FT 份额被吃掉。

The Two Sides of One Mechanism.
memory 不是 agent 的附加品， 是 agent 设计的 核心一部分。

它决定 agent 的"自我"如何连续、如何学习、如何避免重复犯错——也决定它如何被自己累积的错误记忆所污染。
这两面是同一个机制的两面，无法只取其一。
